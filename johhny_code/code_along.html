Code Along — build the crypto-exchange from an empty folder

This is a runbook. Open an empty folder, open this page, and do exactly what each step says, in order. When you reach the end you will have typed — and understood — the entire backend of a working spot cryptocurrency exchange in Go: an in-memory matching engine, a REST API, a WebSocket feed, a settled money layer on SQLite, candlesticks, and a market-maker bot.

How to use this page. Every step has four parts: a WHY WE DO THIS box (the reason a developer reaches for this piece — the concept and the benefit), the file path to create, the code to type, and a walkthrough of what each line does. Type the code yourself (don't paste) — muscle memory is half of learning. Where a real file is long, we type the parts that teach and mark the rest as "same pattern, keep going."
You do not need to know Go yet. Concepts are explained the first time they appear. If a word is new (goroutine, interface, pointer, mutex) it gets a one-paragraph plain-English intro right where you meet it. The only prerequisite is installing Go (Chapter 0).
We build bottom-up — the pieces with no dependencies first (config, the order type), then the matching engine, then the database, then the services on top, and finally main.go that wires it all together. Why this order? Because in Go a package can only use packages that already exist and compile. If you build the engine before the thing it needs, it won't compile and you'll be stuck. Bottom-up means every step compiles the moment you finish typing it. That is the single most important habit for not getting lost in a big codebase.

Chapter 0 — Setup

Before any code, a Go project needs two things: the Go toolchain (the compiler + tools) and a module (a named, versioned bundle that tells Go where your code lives and how to fetch libraries). Without a module, import has no idea what your own packages are called. We create it first so every file we write afterward can find every other file.
STEP 0.1 Install Go & verify.

Download from go.dev/dl (this project needs Go 1.23+). Then, in a terminal:

$ go version
go version go1.23.x ...
STEP 0.2 Create the project & the module. 
$ mkdir crypto-exchange && cd crypto-exchange
$ go mod init github.com/johnny1110/crypto-exchange

That last command writes a go.mod file whose first line is the module name. That name is the prefix for every internal import. So the file engine-v2/model/order.go will be imported elsewhere as github.com/johnny1110/crypto-exchange/engine-v2/model. Pick this name once; changing it later means editing every import.

STEP 0.3 Lay out the folders. In Go, one folder = one package, and the folder structure is the architecture. 
$ mkdir -p engine-v2/model engine-v2/book engine-v2/core engine-v2/market engine-v2/util
$ mkdir -p dto repository/impl service/impl/amm service/serviceHelper
$ mkdir -p controller middleware security ohlcv scheduler ws container settings utils external
$ mkdir -p doc/db_schema
Why so many folders instead of one big one? Three reasons a developer cares about: (1) the compiler enforces layering — a folder can only depend "inward" (the engine can't accidentally import the web layer), so bad coupling literally won't build; (2) change is localised — new storage touches only repository/impl, a new endpoint only controller; (3) you can test a piece in isolation — the engine folder compiles and benchmarks with no database or server, which is how it reaches a million matches/second. Keep this mental picture: outer folders (web) depend on inner folders (engine); never the reverse.
Running go build ./... now succeeds and prints nothing (there's no code yet, but the module is valid). If it errors, your go.mod is missing — re-run step 0.2.

Chapter 1 — Leaf packages: utils & settings

We start at the very bottom of the dependency graph: leaf packages — packages that import (almost) nothing. Everyone can safely import a leaf without creating an import cycle (A imports B imports A, which Go forbids). Two things everything else needs: a tiny math helper + shared counters (utils), and the list of what the exchange trades (settings). Building leaves first means the engine and services can use them immediately.
STEP 1.1 A math epsilon and system counters. utils/counter.go 
package utils

import "sync/atomic"

// Scale is a tiny tolerance for floating-point "dust". Prices/sizes are float64,
// and float math leaves specks (0.30000000000000004). Anything ≤ Scale counts as zero.
const Scale = 0.00000001

// Process-wide counters of orders and trades across ALL users. atomic.AddInt64 is a
// lock-free, thread-safe increment (many goroutines can bump it at once safely).
var ordersPlacedTotal int64
var tradesTotal int64

func IncOrdersPlaced()       { atomic.AddInt64(&ordersPlacedTotal, 1) }
func AddTrades(n int64)      { atomic.AddInt64(&tradesTotal, n) }
func GetOrdersPlaced() int64 { return atomic.LoadInt64(&ordersPlacedTotal) }
func GetTradesTotal() int64  { return atomic.LoadInt64(&tradesTotal) }

Line by line: package utils names the package (matches the folder). const Scale is a compile-time constant. The var ... int64 are package-level variables shared by the whole program. atomic.AddInt64(&x, 1) increments x safely from many goroutines — the & passes the variable's address so the function can change the real thing (Go pointers; more in Chapter 2).

Why a whole package for this? Because both the order service (which increments the counters) and the WebSocket feeder (which reads them for the live "orders/sec" badge) need them, and those two live in different branches of the tree. Put shared state in a leaf and both can import it with zero risk of a cycle. Why atomic and not a plain x++? Because x++ from two goroutines at once can lose an increment (the race from the concepts guide); atomic makes it safe without the cost of a lock.

Chapter 2 — The domain model: the Order

Everything an exchange does revolves around one noun: the order. So the very first "real" thing we build is the Order type and the little words that describe it (buy/sell, limit/market, maker/taker). We put it in engine-v2/model — a package with zero dependencies on the rest of the app (no database, no web) — because the matching engine must be pure and fast, and pure code is easy to test and impossible to accidentally couple to a database.
STEP 2.1 The enums: side, mode, type, status. engine-v2/model/order.go 
package model

import (
    "github.com/johnny1110/crypto-exchange/utils"
    "time"
)

The imports, and why each is here (this is the habit to build):

  • "time" — a Go standard-library package for dates and durations. We need it because every order carries a Timestamp (time.Time) so the book can honour time priority (first-in-first-out at a price). We'll call time.Now() when an order is created.
  • ".../utils" — our own leaf package from Chapter 1. We import it for the utils.Scale epsilon, used to decide when an order's remaining size is "basically zero" (fully filled). Notice we can import it because it's a leaf — no risk of a cycle.

Now the enums. Go has no enum keyword; the idiom is a named integer type plus iota (an auto-incrementing counter that resets each const block):

type Side int
type Mode int
type OrderType int
type OrderStatus string

const ( LIMIT OrderType = iota; MARKET )   // LIMIT=0, MARKET=1
const ( BID  Side = iota; ASK )            // BID=0 (buy), ASK=1 (sell)
const ( MAKER Mode = iota; TAKER )         // MAKER=0, TAKER=1

const (
    ORDER_STATUS_NEW      OrderStatus = "NEW"
    ORDER_STATUS_PARTIAL  OrderStatus = "PARTIAL"
    ORDER_STATUS_FILLED   OrderStatus = "FILLED"
    ORDER_STATUS_CANCELED OrderStatus = "CANCELED"
)
Why make Side its own type instead of using a plain int? Because then the compiler stops you from passing a price where a side is expected — a whole class of bugs vanishes at compile time. Why iota? It's the least-typing way to get a set of related constants; the compiler assigns 0,1,2… automatically. Why is OrderStatus a string and not an int? Because it's written to the database and shown in the API, and a human-readable "FILLED" beats a mysterious 2.
STEP 2.2 The Order struct — the data every order carries. 
type Order struct {
    ID            string
    UserID        string
    Side          Side
    Price         float64
    OriginalSize  float64   // how big the order was when placed
    RemainingSize float64   // how much is still unfilled
    QuoteAmount   float64   // only used by MARKET buy orders (a cash budget)
    Mode          Mode
    FeeRate       float64
    Timestamp     time.Time
}

A struct is just a bundle of named fields — the "record" of the language. Notice there is no Status field: status is computed from the two size numbers, so it can never disagree with reality:

func (o *Order) GetStatus() OrderStatus {
    if o.OriginalSize == o.RemainingSize { return ORDER_STATUS_NEW }
    if o.RemainingSize <= utils.Scale    { return ORDER_STATUS_FILLED }   // <-- why we imported utils
    if o.RemainingSize > 0 && o.RemainingSize < o.OriginalSize { return ORDER_STATUS_PARTIAL }
    return ORDER_STATUS_CANCELED
}

func (o *Order) CounterSide() Side { return o.Side ^ 1 }   // BID↔ASK by flipping the bit

func (o *Order) GetStatus() — the (o *Order) is the receiver: this is a method you call as myOrder.GetStatus(). The * means it receives a pointer to the order (its memory address), not a copy — so it reads the real object and, in other methods, can change it.

STEP 2.3 Constructors and the linked-list node. 
func NewOrder(orderId, userId string, side Side, price, size, quoteAmt float64,
              mode Mode, feeRate float64) *Order {
    return &Order{
        ID: orderId, UserID: userId, Side: side, Price: price,
        OriginalSize: size, RemainingSize: size, QuoteAmount: quoteAmt,
        FeeRate: feeRate, Mode: mode, Timestamp: time.Now(),   // <-- why we imported time
    }
}

// A node in the doubly-linked queue at one price level (built in Chapter 4).
type OrderNode struct {
    Order      *Order
    Prev, Next *OrderNode
}
func (n *OrderNode) Size()  float64 { return n.Order.RemainingSize }
func (n *OrderNode) Price() float64 { return n.Order.Price }

return &Order{...} — the & creates the struct and returns a pointer to it. Returning *Order (a pointer) is deliberate: matching mutates RemainingSize in place, and the linked list stores pointers to these same objects, so everyone sees the same order change.

go build ./engine-v2/model compiles. You now have the atom of the whole system, and you've learned structs, methods, receivers, pointers, iota enums, and the "compute, don't store" trick.

Chapter 4 — The order-book data structures

Before we can match orders we need somewhere to keep them that answers three questions fast: "what's the best price?", "who's next in line at this price?", and "where is order X so I can cancel it?". A developer doesn't reach for a random data structure — each of the three is chosen to make one operation cheap (measured in Big-O: O(1)=constant is great, O(log n) grows slowly, O(n) grows with the whole book and is to be avoided on the hot path). We build three small pieces here, then assemble them into the book in Chapter 5.
STEP 4.1 A FIFO queue per price level (a doubly-linked list / deque). engine-v2/util/deque.go 
package util

import "github.com/johnny1110/crypto-exchange/engine-v2/model"

Why this import: the queue stores *model.OrderNode values (the linked-list node we defined in Chapter 2), so we import our model package to name that type. That's the only thing this file needs from outside.

type OrderNodeDeque struct {
    head, tail *model.OrderNode   // both ends, so push/pop at either end is O(1)
    size       int
    volume     float64            // running sum of remaining sizes at this price
}
// PushBack adds to the tail (newest, back of the line). PopFront removes the head
// (oldest, front of the line). Both are O(1): just re-point a couple of pointers.
// volume is kept in sync on every push/pop so a whole level's size is instant.
Why a doubly-linked list instead of a simple array/slice? Because the hottest operation in the whole exchange is "remove the order at the front of this price level" (a fill). Removing the front of an array means shifting every other element — O(n). Removing the head of a linked list is O(1) — re-point one pointer. At a price level with thousands of resting orders, that's the difference between instant and a stall. And because we keep both head and tail, adding a new order to the back (time priority: newest goes last) is also O(1).
STEP 4.2 One side of the book: a sorted map of price → queue. engine-v2/book/book_side.go 
package book

import (
    "github.com/emirpasic/gods/maps/treemap"
    "github.com/emirpasic/gods/utils"
    "github.com/johnny1110/crypto-exchange/engine-v2/model"
)

The imports, one by one:

  • github.com/emirpasic/gods/maps/treemap — a third-party library ("gods" = Go Data Structures) giving a balanced tree map: a dictionary whose keys stay sorted. We need sorted prices so "best price" is one step away. (Run go get github.com/emirpasic/gods to download it; Go records the exact version in go.sum.)
  • github.com/emirpasic/gods/utils — from the same library, it provides comparator functions like utils.Float64Comparator that tell the tree how to sort float64 keys.
  • .../engine-v2/model — our order/node types again.
type BookSide struct {
    priceLevels      *treemap.Map   // key = price, value = *OrderNodeDeque
    isBid            bool           // bid: best = highest price; ask: best = lowest
    totalVolume      float64        // sum of all resting sizes (for market-order feasibility)
    totalQuoteAmount float64        // sum of size*price
}
// BestPrice(): for a bid, the tree's MAX key; for an ask, its MIN key — O(log n) or better.
// AddOrderNode(price, node): find-or-create the deque at that price, push the node.
// RemoveOrderNode(price, node): unlink it; drop the price level if its deque is now empty.
Why a tree map and not a plain map or a sorted array? A plain hash map (Go's built-in map) has O(1) lookup but is unsorted — finding the best price would mean scanning every key (O(n)). A sorted array keeps order but inserting a new price means shifting elements (O(n)). A balanced tree gives the best of both: O(log n) insert/delete and the min/max (best price) is right there. Prices come and go constantly, so we need cheap insert and cheap "best" — only the tree gives both.
STEP 4.3 An index for O(1) cancel. engine-v2/book/order_index.go 
package book

// no external imports — this file uses only the built-in map type and a small struct.

type indexEntry struct {
    side  model.Side
    price float64
    node  *model.OrderNode
}
type OrderIndex struct {
    index map[string]indexEntry   // orderID -> where that order physically sits
}
Why keep a second structure just for cancels? Because a user cancels by order ID, but the book is organised by price. Without this index, cancelling would mean searching every price level for the ID — O(n) over the whole book (100k+ orders in a busy market). With a hash map from ID straight to the node, cancel is O(1): jump right to it, unlink it. This is a classic space-for-time trade: a little extra memory to make a frequent operation instant.
go get github.com/emirpasic/gods then go build ./engine-v2/... compiles. You've now got the three fast structures; next we make them match orders.

Chapter 5 — The order book & the matching algorithm

This is the heart of the exchange — the code that pairs buyers with sellers. Everything so far was setup for this. We assemble the three structures into an OrderBook, then write the matching loop that honours the two fairness rules (best price first; at the same price, oldest first). This is the code the README benchmarks at ~1–2.7 million orders/second, precisely because Chapter 4 made every step O(1) or O(log n).
STEP 5.1 The book struct + its lock. engine-v2/book/orderbook.go

Every import, and why it's here:

  • .../engine-v2/market — the book belongs to a market, so it holds a *market.MarketInfo.
  • .../engine-v2/model — orders, nodes, sides.
  • .../utils — the Scale epsilon, to decide when a fill is complete.
  • github.com/labstack/gommon/log — a small logging library (the project's chosen logger). We log debug/critical events during matching. (go get github.com/labstack/gommon.)
  • "sync" — standard library concurrency tools. We need sync.Mutex to lock the book so two orders can't interleave and corrupt it (the race condition). This one import is what makes the engine thread-safe.
type OrderBook struct {
    market      *market.MarketInfo
    bidSide     *BookSide
    askSide     *BookSide
    orderIndex  *OrderIndex

}
STEP 5.2 Placement: lock, reject duplicates, route by type. 
func (ob *OrderBook) PlaceOrder(orderType model.OrderType, order *model.Order) ([]Trade, error) {
    if order == nil { return nil, errors.New("order cannot be nil") }
    ob.obMu.Lock()               // take the lock...
    defer ob.obMu.Unlock()       // ...and guarantee we release it however we return

    if ob.orderIndex.OrderIdExist(order.ID) {
        return nil, fmt.Errorf("%w: %s", ErrOrderExists, order.ID)
    }
    switch orderType {
    case model.LIMIT:  return ob.placeLimitOrder(order)
    case model.MARKET: return ob.placeMarketOrder(order)
    default:           return nil, fmt.Errorf("%w: %v", ErrUnsupportedOrderType, orderType)
    }
}
Why Lock() immediately, and why defer Unlock()? Thousands of orders arrive at once on different goroutines. If two touch the same book at the same time, one can read a half-updated state — corruption, lost money. The mutex makes placement serial: one order fully completes before the next starts. defer Unlock() schedules the release for whenever the function returns (success, error, or panic), so you can never forget to unlock and deadlock the exchange. Correctness of money beats cleverness; the single lock is the right call.
STEP 5.3 The maker path (rest) and the taker path (match, then rest). 
func (ob *OrderBook) placeLimitOrder(order *model.Order) ([]Trade, error) {
    if order.Mode == model.MAKER {
        return nil, ob.makeLimitOrder(order)      // maker: just rest, no matching
    }
    trades, err := ob.takeLimitOrder(order)       // taker: match first, rest the remainder
    if err != nil { return nil, err }
    ob.updateLatestPrice(trades)
    return trades, nil
}

// MAKER: wrap in a node, drop at the back of the queue for this price, index it. Done.
func (ob *OrderBook) makeLimitOrder(order *model.Order) error {
    node := &model.OrderNode{Order: order}
    side := ob.getSide(order.Side)
    side.AddOrderNode(order.Price, node)
    ob.addOrderIndex(node)
    return nil
}
STEP 5.4 The matching loop — the single most important function. 
func (ob *OrderBook) takeLimitOrder(order *model.Order) ([]Trade, error) {
    var trades []Trade
    opposite := ob.getOppositeSide(order.Side)          // a buy matches the ASK side

    for order.RemainingSize > 0 {                        // keep going until filled...
        bestPrice, err := opposite.BestPrice()
        if err != nil || !ob.canMatch(order.Side, order.Price, bestPrice) {
            break                                        // ...or nothing crosses anymore
        }
        trade, shouldContinue, err := ob.executeMatch(order, opposite, bestPrice)
        if err != nil { return trades, err }
        trades = append(trades, trade)
        if !shouldContinue { break }
    }
    if order.RemainingSize > 0 {                         // leftover becomes a resting maker
        if err := ob.makeLimitOrder(order); err != nil { return trades, err }
    }
    return trades, nil
}

// canMatch is the price-cross rule: a buyer trades only at asks ≤ its price;
// a seller only at bids ≥ its price.
func (ob *OrderBook) canMatch(side model.Side, orderPrice, bestPrice float64) bool {
    if side == model.BID { return orderPrice >= bestPrice }
    return orderPrice <= bestPrice
}
This for loop is the exchange. Read it as English: "while I still have quantity, look at the best opposite price; if it crosses mine, trade against it; stop when nothing crosses; whatever's left, rest in the book." executeMatch pops the front of the best level (oldest first — time priority), trades the smaller of the two quantities, reduces both, and emits a Trade. Everything else in this whole project is plumbing that feeds orders into this loop and carries its results back out. Type it slowly.
STEP 5.5 Market orders sweep without a price limit. (Type the two functions from Part G.7 of the reference guide — takeMarketBidOrder spends a quote budget; takeMarketAskOrder delivers a size.) They follow the same shape as the loop above but with no canMatch guard — they eat the best rungs until the budget/size runs out, checking first that the book holds enough.

Finally add the snapshot type and RefreshSnapshot (copies the top-20 of each side into an immutable struct) — readers will serve that copy so they never fight this lock. And the trade type:

Chapter 6 — The matching-engine facade

We have one book per market, but the app trades 12 markets. Rather than make the rest of the code juggle 12 books and their locks, we wrap them behind a single friendly door — a facade. The service layer will only ever say "engine, place this order in BTC-USDT"; the facade finds the right book. One simple surface, complexity hidden behind it.
STEP 6.1 engine-v2/core/engine.go 
package core

import (
    "errors"
    "fmt"
    "github.com/johnny1110/crypto-exchange/engine-v2/book"
    "github.com/johnny1110/crypto-exchange/engine-v2/market"
    "github.com/johnny1110/crypto-exchange/engine-v2/model"
    "github.com/labstack/gommon/log"
    "sync"
)

Imports: errors/fmt for errors; book because the engine holds many *book.OrderBook; market for the market list it's built from; model for order types in method signatures; log for debug lines; sync for an RWMutex guarding the map of books (many readers — "which book?" — and rare writers, so a read/write lock beats a plain mutex).

type MatchingEngine struct {
    mu         sync.RWMutex
    orderbooks map[string]*book.OrderBook   // market name -> its book
    markets    []*market.MarketInfo         // kept so we can Reset() to empty books
}

func NewMatchingEngine(markets []*market.MarketInfo) (*MatchingEngine, error) {
    if len(markets) == 0 { return nil, errors.New("market must have at least one market") }
    e := &MatchingEngine{orderbooks: make(map[string]*book.OrderBook, len(markets)), markets: markets}
    for _, m := range markets { e.orderbooks[m.Name] = book.NewOrderBook(m) }
    return e, nil
}

func initDB(testMode bool) (*sql.DB, error) {
    // WAL + synchronous=NORMAL removes the per-commit fsync that otherwise caps throughput.
    dsn := "file:/app/exg.db?_pragma=busy_timeout(10000)&_pragma=journal_mode(WAL)&_pragma=synchronous(NORMAL)"
    db, err := sql.Open("sqlite3", dsn)
    if err != nil { return nil, err }
    db.SetMaxOpenConns(1)          // SQLite is a single writer; one connection serializes writes cleanly
    db.SetMaxIdleConns(1)
    if err := db.Ping(); err != nil { return nil, fmt.Errorf("ping: %w", err) }
    // (ensure ohlcv_1min & credentials tables, seed counters — added in later chapters)
    return db, err
}
Why SetMaxOpenConns(1)? SQLite allows only one writer at a time; giving the pool more connections just makes them fight over the write lock. One connection serialises writes with no contention. Why the WAL pragma? By default SQLite flushes to disk (an fsync) on every commit — safe but slow (~60 orders/sec on a VM). WAL + synchronous=NORMAL skips the per-commit fsync (it only fsyncs at a checkpoint), a ~10× win. These are the kinds of one-line knobs a developer reaches for when a profiler shows the disk, not the code, is the bottleneck.

Chapter 9 — Repositories: SQL behind interfaces

The services shouldn't contain raw SQL — that would weld money logic to table shapes and make it unreadable. So we put all SQL behind repository interfaces: the service says balanceRepo.LockedByUserIdAndAsset(...) ("freeze funds"), and the implementation hides the UPDATE. We deliberately hand-write SQL (no ORM) because an exchange needs precise control over locking and atomic balance moves — exactly what an ORM hides.

STEP 9.2 An implementation — the atomic freeze. repository/impl/balances.go 
package repositoryImpl

import (
    "context"
    "github.com/johnny1110/crypto-exchange/dto"
    "github.com/johnny1110/crypto-exchange/repository"
)

type balanceRepository struct{}
func NewBalanceRepository() repository.IBalanceRepository { return &balanceRepository{} }

func (r *balanceRepository) LockedByUserIdAndAsset(ctx context.Context, db repository.DBExecutor,
        userId, asset string, amt float64) error {
    res, err := db.ExecContext(ctx, `
        UPDATE balances SET available = available - ?, locked = locked + ?
         WHERE user_id = ? AND asset = ? AND available >= ?`,   // <-- the guard
        amt, amt, userId, asset, amt)
    if err != nil { return err }
    n, _ := res.RowsAffected()
    if n == 0 { return errors.New("insufficient balance") }      // nothing matched => not enough
    return nil
}

Imports: "context" (threaded ctx); dto (return types); repository (to name the interface it implements and the DBExecutor type). The constructor returns the interface, not the struct — callers depend on the contract.

Look at AND available >= ?. That single clause makes "check you have enough" and "move the money" one atomic step inside the database. Even if two withdrawals run at the exact same instant, the database will only update a row that still qualifies — so you physically cannot move money you no longer have. This is where the double-spend race from the concepts is actually defeated: at the storage layer, in SQL, not with application-level guessing. The ? placeholders pass values separately from the query text, which is also how you're immune to SQL injection.

Chapter 10 — The transaction helper

Placing an order writes several rows (freeze, insert, trades). If any write fails, they must all undo — you can't half-apply a trade. That's a database transaction: begin, do the writes, commit if all good, roll back if not. Writing begin/commit/rollback by hand at every call site invites the one bug that corrupts balances: a forgotten rollback. So we write the pattern once, as a helper that takes your code as a function.
STEP 10.1 service/impl/txn.go 
package serviceImpl

import (
    "context"
    "database/sql"
)

func WithTx(ctx context.Context, db *sql.DB, fn func(tx *sql.Tx) error) (err error) {
    tx, err := db.BeginTx(ctx, nil)          // open a transaction
    if err != nil { return err }
    defer func() {                           // runs however we leave this function
        if p := recover(); p != nil {        // a panic happened mid-write
            tx.Rollback(); panic(p)
        } else if err != nil {               // fn returned an error
            tx.Rollback()
        } else {                             // clean success
            err = tx.Commit()
        }
    }()
    err = fn(tx)                             // run YOUR writes
    return
}

Imports: "context" for the request handle passed to BeginTx; "database/sql" for *sql.DB/*sql.Tx. Nothing else — this is pure plumbing.

The parameter fn func(tx *sql.Tx) error is the key: in Go, functions are values you can pass around. The caller hands its writes in as fn; WithTx wraps them in the begin/commit/rollback envelope. Because commit/rollback live in a defer, they run no matter how the function exits — normal return, error, or even a panic — so you cannot forget them. That guarantee is the entire reason this helper exists, and you'll wrap every money-write in it. (Named return (err error) lets the deferred code read and even set err to the commit's error.)
You now have the safety net every write path will use. Next we write the services that use it.

Login handed back a token; every later request presents it. Something must map token → which user. That's the credential cache. We back it with a DB table so a token survives a server restart (a memory-only cache would log everyone out on every deploy).

type CredentialCache struct {
    db         *sql.DB
    mu         sync.RWMutex
    tokenCache map[string]*dto.User
}
func (c *CredentialCache) Put(token string, user *dto.User) {
    c.mu.Lock(); c.tokenCache[token] = user; c.mu.Unlock()
    c.db.Exec(`INSERT OR REPLACE INTO credentials(token, user_id) VALUES(?, ?)`, token, user.ID)
}
func (c *CredentialCache) Get(token string) (*dto.User, error) {
    c.mu.RLock(); u, ok := c.tokenCache[token]; c.mu.RUnlock()
    if ok { return u, nil }
    // cache miss (e.g. after restart): rehydrate from the DB
    // ... SELECT user_id FROM credentials, then load the user, cache it, return
}
Why the RWMutex and not a plain map? The map is read on every authenticated request (very frequent) and written only at login (rare). An RWMutex lets many readers proceed at once and only blocks them briefly for a write — the right lock for a read-heavy workload. Why persist to a table at all? So a restart doesn't invalidate every session; on a cache miss we rebuild the user from the durable row. (This exact fix is in the working copy's history.)

func AuthMiddleware(cache *security.CredentialCache) gin.HandlerFunc {
    return func(ctx *gin.Context) {
        token := ctx.GetHeader("Authorization")
        user, err := cache.Get(token)                 // token -> user (Chapter 12)
        if err != nil {
            ctx.AbortWithStatusJSON(http.StatusUnauthorized, gin.H{"code":"..."})
            return                                    // stop the chain; the handler never runs
        }
        ctx.Set("user", user); ctx.Set("userId", user.ID)  // hand the user to the handler
        ctx.Next()                                    // continue to the next link
    }
}
Why return a function that returns a function? The outer function captures a dependency (the cache) once; the inner one runs per request. This "closure captures config" shape is how you give middleware what it needs without globals. ctx.Set puts the resolved user into a per-request bag so the actual handler can read it without re-doing auth. ctx.Abort... stops the chain on a bad token so protected handlers never run unauthenticated.

Chapter 16 — OHLCV: turning trades into candlesticks

A price chart needs candlesticks — open/high/low/close per time bucket. Trades arrive as a fast stream; the chart wants tidy 1-minute (etc.) bars. So we build an aggregator: trades flow into a channel, a background worker folds each trade into the current bucket for every interval, and finished bars are flushed to the DB in batches. We decouple "a trade happened" from "a candle got written" with a channel so a burst of trades never blocks the matching path.

    }
}

Imports: just "sync" (if you guard subscriber lists). The core idea is the buffered chan *Trade. The select { case ch<-t: default: } is a non-blocking send: if the buffer is full we drop the trade for charting rather than stall the order that produced it. Charts may skip a tick; orders may never wait.

Imports, why: "context" to stop the loop on shutdown; "sync" to guard per-symbol candle state; "time" to compute bucket boundaries and drive the flush ticker; utils for price rounding. Why aggregate in memory and flush in batches instead of writing every trade? Because writing a row per trade would hammer the single SQLite writer — the same bottleneck we tuned in Chapter 8. Folding trades into a handful of in-memory bars and flushing periodically turns thousands of trades into a few writes. This is the classic "batch the writes" pattern, and it's why the chart keeps up under load. Chapter 17 — WebSocket: pushing live data to browsers

REST is pull: the browser must ask, repeatedly, for fresh prices. That's wasteful and laggy. WebSocket is push: the client connects once and the server streams updates. We build a hub — one goroutine that owns the set of clients and fans messages out over channels — so no two goroutines ever touch the client map at once (no locks, no races). Public channels (orderbook, ohlcv, markets, sysstats) broadcast to all; private channels (user_data) go only to the owning token.
STEP 17.1 The client & the hub. ws/client.go, ws/hub.go 
import (
    "context"
    "github.com/gorilla/websocket"
    "sync"
)

Imports, why: github.com/gorilla/websocket — the WebSocket implementation (upgrades an HTTP request to a socket; go get github.com/gorilla/websocket); "context" to stop the hub on shutdown; "sync" only where a helper map needs guarding. The hub itself avoids locks by owning state in one goroutine.

type Client struct {
    conn         *websocket.Conn
    Send         chan []byte          // buffered outbound queue for THIS client (buffer 1024)
    subscriptions map[string]bool     // which channels it wants
    token        string               // "" until it authenticates for user_data
}
type Hub struct {
    clients    map[*Client]bool
    register   chan *Client
    unregister chan *Client
    broadcast  chan *BroadcastMsg
}
func (h *Hub) Run(ctx context.Context) {
    for {
        select {                                  // ONE goroutine owns the client set
        case c := <-h.register:   h.clients[c] = true
        case c := <-h.unregister: delete(h.clients, c); close(c.Send)
        case m := <-h.broadcast:
            for c := range h.clients {
                if c.wants(m.Channel, m.Token) {  // public OR its own private data
                    select { case c.Send <- m.Data: default: /* slow client: drop */ }
                }
            }
        }
    }
}
Why funnel everything through one select loop instead of locking the client map? Because the map is mutated (connect/disconnect) and read (broadcast) constantly. A mutex would serialise all of it and invite deadlocks. Instead, the hub is the only goroutine that touches the map; everyone else sends it a message on a channel. That's the "share memory by communicating" philosophy — concurrency without locks. Each client has its own buffered Send channel, and a non-blocking send means one slow browser can't stall the broadcast to everyone else (we raised this buffer 256→1024 so bursty market data doesn't drop). The per-client token is the whole public/private split: a message tagged with a token only reaches the matching client.
STEP 17.2 The channel model. ws/model.go — define the channel names the front-end subscribes to: ORDERBOOK, OHLCV, MARKETS, SYSSTATS (public), and USER_DATA (private, needs a token). Each subscribe request carries params (e.g. {market: "BTC-USDT"}). Watch this exact bug: the request param key must match what the backend reads — the front-end sending {symbol:...} while the backend reads {market:...} silently yields an empty market and no data. Names are a contract; keep both sides identical.

Chapter 18 — Schedulers: the heartbeats

Some work must happen on a clock, not per-request: snapshot the order book every 300ms, push WS frames every 400ms–1s, refresh 24h market stats every 30s, top up AMM liquidity every 5 min. Each is a scheduler — a goroutine with a time.Ticker. Centralising them keeps timing in one place and off the request path.
STEP 18.1 The WS data feeder (the one that makes the UI feel alive). scheduler/ws_data_feeder.go

Every import is one data source this job stitches into WebSocket frames.

func (j *WSDataFeederJob) Start(ctx context.Context) {
    fast := time.NewTicker(400 * time.Millisecond) // orderbook: feels instant
    slow := time.NewTicker(1 * time.Second)        // ohlcv, markets, sysstats, user_data
    for {
        select {
        case <-ctx.Done(): return
        case <-fast.C:  j.pushOrderbooks()          // broadcast to ORDERBOOK subscribers
        case <-slow.C:
            j.pushOHLCV(); j.pushMarkets()
            j.pushSysStats(utils.SnapshotStats())   // orders/sec, trades/sec, totals, duration
            j.pushUserData()                        // per-token private orders+balances
        }
    }
}
Why two tickers at different rates? The order book changes every match — 400ms makes it feel live. Candles, 24h stats and counters change slowly — 1s is plenty and saves work. Why push from a scheduler instead of on every trade? Because at hundreds of trades/sec you'd flood browsers; a fixed cadence coalesces many changes into one frame. This is the "snapshot + interval push" model borrowed from real exchanges: the UI shows the latest state at a steady beat, not every micro-event. The order-book snapshot itself is produced by a separate 300ms scheduler and cached, so the feeder just forwards the freshest cached copy — cheap.

Chapter 19 — AMM: a bot that provides liquidity

An empty order book is a dead market — no one can trade if no one is quoting. Chapter 20 — The container: the composition root

Every layer we built depends only on interfaces, never on concrete types — that's what kept them testable and decoupled. But someone has to create the real objects and hand each layer its dependencies. That someone is the container: the single place ("composition root") where the whole graph is constructed, bottom-up. Nothing else in the codebase does new X() for a service; they receive what they need. Change an implementation? You edit one file.

Chapter 21 — main.go: boot order & recovery

main() is the ignition. Its job is a precise sequence: open the DB, build the engine, recover the order book from disk (open orders survive a restart), build the container, start schedulers, mount routes, listen. STEP 21.1 main.go (package main)

func main() {
    db, err := initDB(true)                       // Chapter 8 (true = seed schema+mock data)
    if err != nil { log.Fatal(err) }

    engine := core.NewMatchingEngine()            // Chapter 6
    for _, m := range settings.ALL_MARKETS {      // register every market's order book
        engine.AddMarket(m)
    }
    recoverOrderBook(db, engine)                  // reload OPEN orders from DB into memory

    c := container.NewContainer(db, engine)       // Chapter 20 — build everything
    c.StartSchedulers()                           // heartbeats on (Chapter 18)
    defer c.Cleanup()                             // close DB on exit

    router := gin.New()
    RegisterRoutes(router, c)                     // Chapter 15
    router.Run(":3000")                           // listen
}
Why recoverOrderBook before serving? The engine lives in memory; a restart would otherwise "lose" every resting order even though it's safe in the orders table. So on boot we SELECT all still-open orders and replay them into the in-memory book — now memory and disk agree before the first request lands. This is the reconciliation step that makes an in-memory engine safe to restart. The defer c.Cleanup() plus the signal catch is the mirror image: on the way down, we close the DB cleanly so WAL is checkpointed and nothing is half-written.

STEP 22.3 Prove conservation of money.
The one test that matters for an exchange: total funds before == total funds after a flurry of trades. Sum every user's available + locked across all assets, run N random orders, sum again. If the numbers differ by a cent, a settlement path is wrong. This invariant is why we built two-phase placement inside WithTx (Chapters 10–11) — here's where you cash that in.
STEP 22.4 Push throughput — and know the ceiling.
The matching engine (pure memory, Chapters 4–6) does ~1M ops/sec. But end-to-end each order also commits to SQLite, and SQLite is a single writer — so real throughput lands around ~500–630 orders/sec on a good machine, no matter how many CPU cores you throw at it (the backend pegs exactly one core on the write path). We proved this: swapping the SQLite driver made it slower; adding a connection pool made it slower (write-lock contention); an 18-core box matched an 8-core one. The honest conclusion — and the real engineering lesson — is that your bottleneck is the durable writer, not the algorithm. To go faster you'd batch commits, shard the DB, or move to a write-ahead log service — a different project. For a self-contained, restart-safe demo, the single-file SQLite ceiling is the right trade.
