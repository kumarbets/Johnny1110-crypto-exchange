Build From Scratch — johnny1110/crypto-exchange

This guide takes you from an empty folder to a fully working spot cryptocurrency exchange written in Go — the exact project that lives in crypto-exchange/, with a Vue front-end in crypto-exchange-fe/. When you finish you will have built:

  • An in-memory limit-order matching engine (millions of orders/sec) with a proper price—time-priority order book.
  • A REST API (Gin) for register / login / balances / place & cancel orders / market data.
  • A WebSocket server pushing live order-book, candlestick and market feeds.
  • A transactional settlement layer on SQLite that freezes funds, records trades and applies fees atomically.
  • An OHLCV (candlestick) aggregator that turns the trade stream into 15m/1h/4h/1d/1w bars.
  • A background Automated Market Maker (AMM) bot that quotes liquidity on all 12 markets from a real index price.
  • A set of schedulers that refresh snapshots, compute 24h stats, run the AMM and feed WebSocket clients.

Part A — Go from absolute zero

Everything below is a concept the exchange actually uses. Each section ends with a "where you meet it" pointer so the abstract idea has a concrete home.

A.1 — What Go is & why this project uses it

Go (a.k.a. Golang) is a small, statically-typed, compiled language from Google. It has three properties that make it ideal for an exchange:

  • Fast, native binaries. The whole exchange compiles to a single ~20 MB executable with no runtime to install. The order-book benchmarks in the README hit ~1.9 million orders/second — that is Go compiling to machine code.
  • Cheap concurrency. A goroutine costs a few KB, so the app happily runs dozens of them: an HTTP server, a WebSocket server, four schedulers, an OHLCV worker pool — all at once.
  • Boring on purpose. One way to write a loop, one error convention, one formatter (gofmt). Large codebases stay readable.

A.4 — Variables, constants, types, zero values

var price float64 = 65000.0   // explicit type
size := 0.5                   // := infers float64
const feeRate = 0.002         // constant, untyped

var status string             // zero value: "" (empty string)
var filled bool               // zero value: false
var count int                 // zero value: 0
var book *OrderBook           // zero value: nil (pointer)

Key idea: an uninitialised variable is never "garbage" — it holds the zero value for its type. This is why a freshly-created Order whose RemainingSize field is never set is 0, and a *OrderBook you forgot to build is nil (and dereferencing it panics).

A.5 — iota enums

Go has no enum keyword. Instead you declare a named integer type and use iota (an auto-incrementing counter, resets per const block). This is how the exchange models order sides and types — here is the real code:

// engine-v2/model/order.go
type Side int
type Mode int
type OrderType int
type OrderStatus string

const ( LIMIT OrderType = iota; MARKET )       // LIMIT=0, MARKET=1
const ( BID Side = iota; ASK )                 // BID=0,   ASK=1
const ( MAKER Mode = iota; TAKER )             // MAKER=0, TAKER=1

const (
    ORDER_STATUS_NEW      OrderStatus = "NEW"
    ORDER_STATUS_PARTIAL  OrderStatus = "PARTIAL"
    ORDER_STATUS_FILLED   OrderStatus = "FILLED"
    ORDER_STATUS_CANCELED OrderStatus = "CANCELED"
)

Because Side is its own type (not a bare int), you cannot accidentally pass a price where a side is expected — the compiler rejects it. A neat trick the engine uses: the opposite side is just a bit-flip, o.Side ^ 1 (BID↔ASK).

A.6 — Functions & the (value, error) idiom

Functions can return multiple values. Go's universal convention is "result, error": the last return is an error that is nil on success.

func (e *MatchingEngine) GetOrderBook(market string) (*book.OrderBook, error) {
    ob, ok := e.orderbooks[market]
    if !ok {
        return nil, fmt.Errorf("market %s not found", market)
    }
    return ob, nil
}

// caller:
ob, err := engine.GetOrderBook("BTC-USDT")
if err != nil {
    return nil, err   // bubble it up
}
// ... use ob

You will type if err != nil { return ... } hundreds of times. That verbosity is deliberate: every failure is visible at the call site.

A.7 — Control flow: if / for / switch

for is Go's only loop keyword (it covers while, for, and infinite loops). switch needs no break. The matching loop is a real example:

// keep matching until the order is filled or no price crosses
for order.RemainingSize > 0 {
    bestPrice, err := opposite.BestPrice()
    if err != nil || !ob.canMatch(order.Side, order.Price, bestPrice) {
        break
    }
    // ... execute one match
}

switch orderType {
case model.LIMIT:  return ob.placeLimitOrder(order)
case model.MARKET: return ob.placeMarketOrder(order)
default:           return nil, ErrUnsupportedOrderType
}

A.8 — Slices, arrays & maps

A slice is a growable list ([]T); a map is a hash table (map[K]V). Both appear everywhere:

trades := make([]Trade, 0, len(orderDTOs))  // slice, pre-sized capacity
trades = append(trades, trade)              // grow it

orderbooks := make(map[string]*book.OrderBook)   // market name -> book
ob, ok := orderbooks["BTC-USDT"]                 // comma-ok: ok is false if absent
if ok { /* found */ }

The value, ok := m[key] "comma-ok" form is how you tell "missing key" from "present but zero". The engine uses exactly this in GetOrderBook above.

A.9 — Structs & methods

A struct groups fields; a method is a function with a receiver. The Order struct is the heart of the engine:

// engine-v2/model/order.go
type Order struct {
    ID            string
    UserID        string
    Side          Side
    Price         float64
    OriginalSize  float64
    RemainingSize float64
    QuoteAmount   float64   // only used by MARKET bid orders
    Mode          Mode
    FeeRate       float64
    Timestamp     time.Time
}

// a method: (o *Order) is the receiver
func (o *Order) GetStatus() OrderStatus {
    if o.OriginalSize == o.RemainingSize { return ORDER_STATUS_NEW }
    if o.RemainingSize <= utils.Scale    { return ORDER_STATUS_FILLED }
    if o.RemainingSize > 0 && o.RemainingSize < o.OriginalSize { return ORDER_STATUS_PARTIAL }
    return ORDER_STATUS_CANCELED
}

Notice status is computed from sizes rather than stored — a single source of truth. utils.Scale is a tiny epsilon (0.00000001) so floating-point dust counts as "filled".

A.10 — Pointers (why *Order and &o)

A pointer holds the address of a value. &o takes an address; *p dereferences it. You use a pointer when a method must mutate the value, or to avoid copying a large struct.

func NewOrder(...) *Order {         // returns a pointer to a new Order
    return &Order{ ID: orderId, ... }
}

func (o *Order) fill(qty float64) { // pointer receiver => mutation sticks
    o.RemainingSize -= qty
}

The matching engine threads *Order everywhere precisely because matching reduces RemainingSize in place, and the doubly-linked list stores pointers to the same order objects.

Rule of thumb: use a pointer receiver if the method changes the struct or the struct is big; use a value receiver for small, read-only helpers. The exchange uses pointer receivers almost everywhere for consistency.

A.11 — Interfaces

An interface is a set of method signatures. Any type that has those methods satisfies the interface automatically — no implements keyword. This is the single most important idea for understanding this codebase, because every layer is wired through interfaces.

// repository/interface.go (shape)
type IOrderRepository interface {
    Insert(ctx context.Context, db DBExecutor, order *dto.Order) error
    CancelOrder(ctx context.Context, db DBExecutor, orderId string, remainingSize float64) error
    PaginationQuery(...) ([]*dto.Order, error)
    // ...
}

The service layer depends on IOrderRepository (the interface), not on the concrete SQLite implementation. That means you can swap SQLite for Postgres, or a fake in a unit test, without touching the service. Two patterns you will see constantly:

  • Constructor returns the interface: func NewOrderRepository() IOrderRepository { return &orderRepository{} } — callers only ever see the interface.
  • An abstraction that lets a repo run on either a DB or a transaction: DBExecutor is an interface satisfied by both *sql.DB and *sql.Tx, so the same Insert works inside or outside a transaction.

A.12 — Generics (Go 1.18+)

Generics let a function or type work over many element types with compile-time safety. The order book's per-price-level list and the gods TreeMap are generic containers:

func Max[T int | float64](a, b T) T {
    if a > b { return a }
    return b
}
Max(3, 5)        // T inferred as int
Max(1.2, 0.7)    // T inferred as float64

You mostly consume generics here (via the emirpasic/gods TreeMap and the project's own OrderNodeDeque) rather than write them.

A.13 — Errors: return them, don't throw

Go has no exceptions for ordinary failures. error is just an interface with one method, Error() string. You create sentinel errors once and compare with errors.Is:

// service/impl/orders.go
var (
    ErrInvalidInput        = errors.New("invalid input")
    ErrOrderNotFound       = errors.New("order not found")
    ErrInsufficientBalance = errors.New("insufficient balance")
)

if err := s.balanceRepo.LockedByUserIdAndAsset(...); err != nil {
    return ErrInsufficientBalance   // map low-level failure to a domain error
}

panic exists but is reserved for truly unrecoverable startup faults (e.g. a scheduler that fails to start) — you will see it only in main.go/startup_helper.go.

A.14 — Concurrency: goroutines, channels, mutex, sync.Map

Start concurrent work with go f(). Communicate over channels. Protect shared memory with a mutex. All four appear in the exchange:

// goroutine: the WebSocket server runs alongside the HTTP server
go func() {
    http.HandleFunc("/ws", ...)
    http.ListenAndServe(":8081", nil)
}()

// channel: the OHLCV aggregator streams trades between goroutines
tradeCh chan *Trade

// RWMutex: many readers OR one writer guards the engine's book map
type MatchingEngine struct {
    mu         sync.RWMutex
    orderbooks map[string]*book.OrderBook
}

// sync.Map: lock-free concurrent map of symbol -> realtime bars
realtimeSymbolBars sync.Map

Rule: sync.RWMutex when reads vastly outnumber writes (order-book snapshots), a plain sync.Mutex for the book itself during matching, and sync.Map for the OHLCV per-symbol state that many goroutines touch.

A.15 — Packages & visibility

Each folder is one package. Capitalised identifiers are exported (public to other packages); lowercase ones are private. So Order, PlaceOrder are usable elsewhere; orderService, makeLimitOrder are internal helpers. This convention is the access control — there is no public/private keyword.

// defer + transaction pattern used by every write path
func WithTx(ctx context.Context, db *sql.DB, fn func(tx *sql.Tx) error) error {
    tx, err := db.Begin()
    if err != nil { return err }
    defer tx.Rollback()          // no-op if we Commit first; safety net on any early return
    if err := fn(tx); err != nil { return err }   // fn does the real work
    return tx.Commit()
}

A.20 — The four Go design patterns this codebase leans on

Recognising these four makes the whole codebase readable — they recur in almost every package.

1. Constructor returns an interface

Every layer exposes a New… function that returns an interface, hiding the concrete struct. Callers depend on the contract, never the implementation — that is what makes the layers swappable and testable.

func NewIOrderService(db *sql.DB, engine *core.MatchingEngine, /* repos… */) service.IOrderService {
    return &orderService{ db: db, engine: engine, /* … */ }   // returns the INTERFACE
}

2. Builder (fluent construction)

A DTO with many fields is assembled step-by-step with chained methods, ending in Build(). Reads like a sentence and avoids a 12-argument constructor. Used for orders.

order := dto.NewOrderBuilder().
    Market("BTC-USDT").User(userID).Side(model.BID).
    Price(65000).Size(0.5).Mode(model.MAKER).
    Build()

3. Strategy (behaviour chosen at runtime)

Instead of a big if orderType == LIMIT … else …, the service picks a strategy object that knows how to place that kind of order. Adding a new order type means adding a struct, not editing a switch.

type OrderPlacementStrategy interface {
    Execute(ctx, service, orderCtx) error
}
// LimitOrderStrategy and MarketOrderStrategy each implement Execute();
// getOrderPlacementStrategy(req) returns the right one.

4. Closure callback (a function is a first-class value)

The transaction helper takes your code as a function argument and runs it inside a begin/commit/rollback envelope. You never write transaction boilerplate again — the pattern owns commit-on-success, rollback-on-error.

WithTx(ctx, s.db, func(tx *sql.Tx) error {   // <-- a function passed as data
    // … all your writes here …
    return nil                                // nil => commit; non-nil => rollback
})

(A fifth, the Facade, you already met: MatchingEngine is a thin front over many OrderBooks, giving the rest of the app one simple door to knock on.)

B.5 — Step 3: the matching algorithm

This is the core. PlaceOrder locks the book, rejects duplicate IDs, then dispatches on order type. The rules:

  • LIMIT + MAKER → rest in the book, no matching.
  • LIMIT + TAKER → match against the opposite side while prices cross; rest any remainder.
  • MARKET → sweep the opposite side (never rests); bid consumes by quote-amount, ask by size.
STEP 3 engine-v2/book/orderbook.go 
func (ob *OrderBook) PlaceOrder(orderType model.OrderType, order *model.Order) ([]Trade, error) {
    if order == nil { return nil, errors.New("order cannot be nil") }
    ob.obMu.Lock()
    defer ob.obMu.Unlock()

    if ob.orderIndex.OrderIdExist(order.ID) {
        return nil, fmt.Errorf("%w: %s", ErrOrderExists, order.ID)
    }
    switch orderType {
    case model.LIMIT:  return ob.placeLimitOrder(order)
    case model.MARKET: return ob.placeMarketOrder(order)
    default:           return nil, fmt.Errorf("%w: %v", ErrUnsupportedOrderType, orderType)
    }
}

func (ob *OrderBook) placeLimitOrder(order *model.Order) ([]Trade, error) {
    if order.Mode == model.MAKER {          // maker: just rest it
        return nil, ob.makeLimitOrder(order)
    }
    trades, err := ob.takeLimitOrder(order) // taker: match, then rest remainder
    if err != nil { return nil, err }
    ob.updateLatestPrice(trades)
    return trades, nil
}
the taker loop — match while the price crosses 
func (ob *OrderBook) takeLimitOrder(order *model.Order) ([]Trade, error) {
    var trades []Trade
    opposite := ob.getOppositeSide(order.Side)

    for order.RemainingSize > 0 {
        bestPrice, err := opposite.BestPrice()
        if err != nil || !ob.canMatch(order.Side, order.Price, bestPrice) {
            break   // no counter-order, or the spread no longer crosses
        }
        trade, shouldContinue, err := ob.executeMatch(order, opposite, bestPrice)
        if err != nil { return trades, err }
        trades = append(trades, trade)
        if !shouldContinue { break }
    }
    if order.RemainingSize > 0 {   // leftover becomes a resting maker order
        if err := ob.makeLimitOrder(order); err != nil { return trades, err }
    }
    return trades, nil
}

canMatch is the price-crossing test: a BID matches when orderPrice >= bestAsk; an ASK matches when orderPrice <= bestBid. executeMatch pops the best resting order, trades min(incoming.Remaining, resting.Remaining), reduces both, emits a Trade, and puts the resting order back at the head of its level if it still has size. Market orders (takeMarketBidOrder/takeMarketAskOrder) first check the opposite side has enough totalQuoteAmount/totalVolume, then sweep it.

Why it's fast. Best-price access is O(1) via the TreeMap's min/max, matching pops the front of a linked list (O(1)), and cancels are O(1) through the index. That is how the README benchmarks reach ~1.9M orders/sec.
Worked example — watch one taker sweep the book

Say the ask side of BTC-USDT currently rests these maker orders (price—time priority: lowest price first, oldest first within a price):

ASK ladder (sellers)          a limit BUY arrives:
  65,050  ×0.4  (M3)            side=BID  price=65,020  size=0.7  mode=TAKER
  65,020  ×0.2  (M2, older)
  65,020  ×0.3  (M1, newer)     canMatch(BID, 65,020, bestAsk) ?  65,020 >= 65,020 ✓
  1. Best ask = 65,020. It crosses (buy price ≥ best ask), so match. Pop M2 (oldest at 65,020). Trade min(0.7, 0.2)=0.2 @ 65,020. Incoming remaining = 0.5.
  2. Best ask still 65,020 (M1). Trade min(0.5, 0.3)=0.3 @ 65,020. Incoming remaining = 0.2. The 65,020 level is now empty and dropped.
  3. Best ask = 65,050. Buy price 65,020 < 65,050 → canMatch is false. Stop.
  4. Remaining 0.2 > 0, so the leftover becomes a resting bid at 65,020 via makeLimitOrder.

Result: two Trade records (0.2 and 0.3), latestPrice=65,020, and a new best bid of 0.2 @ 65,020. A market buy would behave the same but with no price guard — it keeps sweeping until its quote-amount is spent or the ask side runs dry.

PlaceOrder runs in two phases. Phase 1 is a single DB transaction: freeze funds, insert the order, run the match, persist trades, and fan the trades into the candlestick stream. Phase 2 settles balances/fees for the resulting trades.

Phase 1 — placement (atomic) 
func (s *orderService) executeOrderPlacementPhase(ctx context.Context, orderCtx *dto.PlaceOrderContext, isMarketOrder bool) error {
    return WithTx(ctx, s.db, func(tx *sql.Tx) error {
        // 1. Freeze user funds (available -> locked). Fails => insufficient balance.
        if err := s.balanceRepo.LockedByUserIdAndAsset(ctx, tx,
                orderCtx.UserID, orderCtx.Assets.FreezeAsset, orderCtx.Assets.FreezeAmt); err != nil {
            return ErrInsufficientBalance
        }
        // 2. Insert the order row (status NEW).
        if err := s.orderRepo.Insert(ctx, tx, orderCtx.OrderDTO); err != nil { return UnknownError }

        // 3. Run it through the matching engine.
        engineOrder := serviceHelper.NewEngineOrderByOrderDto(orderCtx.OrderDTO)
        trades, err := s.engine.PlaceOrder(orderCtx.Market, orderCtx.Request.OrderType, engineOrder)
        if err != nil { return err }

        // 4. Reflect engine result back onto the DTO (remaining size, avg price).
        orderCtx.SyncTradeResult(engineOrder, trades)

        // 5. Persist trades + feed the candlestick stream.
        if len(orderCtx.Trades) > 0 {
            if err := s.tradeRepo.BatchInsert(ctx, tx, trades); err != nil { return UnknownError }
            for _, trade := range trades {
                s.klineTradeStream.SyncTrade(&ohlcv.Trade{
                    Symbol: trade.Market, Price: trade.Price, Volume: trade.Size, Timestamp: trade.Timestamp})
            }
        }
        // 6. Market-BID orders: rewrite original_size to what was actually bought.
        if isMarketOrder && orderCtx.Request.Side == model.BID { /* UpdateOriginalSize */ }
        return nil
    })
}

Because it is wrapped in WithTx, if any step fails the whole thing rolls back — the freeze, the order row and the trades all disappear together. That is the atomicity guarantee an exchange must have.

B.10 — Step 8: settlement (fees, balance moves)

Phase 2 takes the trades and moves money for real. serviceHelper.ProcessTradeSettlement computes, per trade, how much each side's locked balance to release, how much of the counter-asset to credit, and the fee to deduct; the service then applies those diffs:

STEP 8 service/impl/orders.go → serviceHelper/settlement.go 
func (s *orderService) executeTradeSettlementPhase(ctx context.Context, orderCtx *dto.PlaceOrderContext) error {
    if len(orderCtx.Trades) == 0 { return nil }
    settlementResult, err := serviceHelper.ProcessTradeSettlement(orderCtx)
    if err != nil { return err }

    for _, u := range settlementResult.OrderUpdates {          // reduce remaining, add fees
        s.orderRepo.SyncTradeMatchingResult(ctx, s.db, u.OrderID,
            u.RemainingSizeDecreasing, u.DealtQuoteAmountIncreasing, u.FeesIncreasing)
    }
    for userID, settlement := range settlementResult.UserSettlements {  // credit/debit assets
        s.updateUserAssets(ctx, s.db, userID, orderCtx.Assets, settlement)
    }
    return nil
}

Fees are asymmetric: makers pay less than takers (see the seeded users — maker 0.1%, taker 0.2%), captured per side on each Trade. Cancelling an order runs the mirror operation: remove from the engine, set status CANCELED, and unlock the frozen balance.

With initDB(false) the server uses it as-is, so you get a fully populated exchange on first boot. Ganache/Ethereum is not required — the on-chain client is defined but never called in the hot path.

Option A — Docker (the documented path)

The Dockerfile builds a static binary and copies the seeded DB into /app:

$ cd crypto-exchange
$ docker build -t exg .
$ docker run -d --name exg -p 8080:8080 -p 8081:8081 exg
$ curl http://localhost:8080/health          # {"status":"ok"}
$ curl http://localhost:8080/api/v1/markets  # 12 markets with live prices

Option B — Local Go

The DB path is hard-coded to /app/exg.db, so create that path once (or run via Docker). On Linux/macOS:

$ sudo mkdir -p /app && sudo cp app/exg.db /app/exg.db && sudo chown $USER /app/exg.db
$ go run .            # or: make build && ./bin/exchange
# REST on :8080, WebSocket on :8081
Fresh, empty DB? To rebuild schema + seed data from SQL on every boot, temporarily switch initDB(false) to initDB(true) in main.go — it executes doc/db_schema/schema.sql then testing_data.sql.

C.2 — Run the Vue front-end

The front-end is Vue 3 built with vue-cli-service (not Vite). It talks to the backend through one file. Point that file at wherever your backend runs:

Open the URL it prints. You should see the markets table populate, an order book with AMM depth, a candlestick chart (lightweight-charts), and a login modal. Register a new user or log in as one of the seeded accounts, then place an order and watch it settle live.

D.2 — The throughput story: 60 → 628 orders/sec, and why not 5000

The README benchmarks show the matching engine doing ~1–2.7 million orders/sec in memory. Yet the first end-to-end run (real HTTP → service → SQLite → settlement) measured only ~60 orders/sec. That gap is the whole lesson.

Where the time actually goes

The engine is in-memory (nanoseconds). The cost is durability: every order is persisted. With SQLite's defaults (journal_mode=DELETE, synchronous=FULL) every commit performs an fsync() — a physical flush to disk. Worse, an order that trades writes in two phases, and the settlement phase issues several auto-commit statements, so a single order can trigger multiple fsyncs. On the VM's disk an fsync is ~8 ms, so ~2 fsyncs/order ≈ 16 ms ≈ ~60 orders/sec. The bottleneck was the disk, not the code.

The fix: WAL + synchronous=NORMAL

// crypto-exchange/db.go  — the one line that mattered
dsn := "file:/app/exg.db?_pragma=busy_timeout(10000)&_pragma=journal_mode(WAL)&_pragma=synchronous(NORMAL)"
db, err := sql.Open("sqlite3", dsn)
  • WAL (Write-Ahead Logging): commits append to a -wal side-file instead of rewriting the main DB; readers never block the writer.
  • synchronous=NORMAL: stops fsync-ing on every commit — it only fsyncs at a periodic checkpoint. This is the big win. (Trade-off: a crash at the wrong instant can lose the last few commits — acceptable for a demo, tunable for production.)

Result: ~628 orders/sec — a clean 10×, verified with a 20,000-order load test.

Why a connection pool made it worse (the instructive failure)

The obvious next idea — "use many DB connections to go parallel" — we tried, with MaxOpenConns(12) and BEGIN IMMEDIATE. It measured slower (~366/sec) with zero errors. The reason is fundamental: SQLite is a single-writer database. Only one write transaction commits at a time; twelve connections just queue on the same write lock and add contention. We reverted to a single connection.

In-memory engine only (benchmark, no persistence)~1,000,000–2,700,000

Why we did not force 5,000+ orders/sec. Getting there is not a tuning knob — it needs an architectural change to a working repo we were asked to keep intact: e.g. batching many orders per transaction, moving settlement off the request path (async), or replacing single-writer SQLite with a server database (Postgres) or a sharded/append-only design. Each is a real redesign with correctness implications for money. The honest result is: WAL takes this design from 60 to ~630 orders/sec; beyond that you change the persistence architecture, not a setting. ~630 orders/sec is already a firehose in the UI (hundreds of trades/second).

D.3 — Making live high-frequency trading visible

Two things surprised us in the UI, both now understood:

  1. The order book and candles updated, but "Open Orders" and "Order History" stayed empty. Those two panels show only the logged-in user's own orders. If you are logged in as an account that has not traded, they are correctly empty. Log in as an account the generator is trading on and they fill instantly.
  2. They didn't refresh live. The component's startAutoRefresh only re-polled the order book each second. We extended it (guarded by login) to also refresh the user's orders, history and balances:
// crypto-exchange-fe/src/components/MarketOrderBook.vue
startAutoRefresh() {
  this.refreshInterval = setInterval(() => {
    this.fetchOrderBook()
    if (authUtils.isAuthenticated()) {
      this.fetchOpenOrders()     // <-- added
      this.fetchClosedOrders()   // <-- added (trade history)
      this.fetchBalances()       // <-- added
    }
  }, 1000)
}

Now, logged in, you watch open orders appear and get consumed and the trade history scroll — once per second — while the generator runs. Note this is polling, not push; the order book deliberately polls the REST snapshot (robust and simple), while the candlestick chart uses the WebSocket ohlcv feed.

D.4 — The load-test harness & the 5-user HFT demo

To drive thousands of real orders we wrote a ~60-line standalone Go program (outside the repo module). It logs in as a user, then fires orders concurrently through the real REST API — a worker pool of goroutines, each POSTing to /api/v1/orders/BTC-USDT — and reports accepted/sec and how many trades resulted. It places a tight price band around the mid so most orders cross and match:

// pseudo-shape of the harness (not part of the app)
jobs := make(chan int, n)
for i := 0; i < concurrency; i++ {           // a pool of worker goroutines
    go func() {
        for range jobs {
            side  := rng.Intn(2)              // buy / sell
            off   := float64(rng.Intn(6) - 3) // tight band => lots of crossing
            mode  := maker; if aggressive(side, off) { mode = taker }
            POST("/api/v1/orders/BTC-USDT", {side, order_type:0, mode, price:mid+off, size:0.001})
        }
    }()
}
// measure wall-clock over n orders => orders/sec

For the live demo we created five funded users and run one generator per user (as systemd units on the VM), so orders come from all five and cross-match against each other — realistic multi-party trading rather than self-trading. Aggregate throughput is still the single-writer ceiling (~630/sec), now spread across five accounts, each seeing its own share of ~130 orders/sec.

D.6 — WebSocket-first data flow (public + private channels)

The UI originally polled REST every second (order book and the per-user orders/trades/balances we added), which is a lot of chatter. We moved everything to WebSocket. The important insight is a public vs private split:

DataChannelTypeKeyed by
Candlesticksohlcvpublicsymbol + interval
Order book (top-20 snapshot)orderbookpublicmarket
All-markets summarymarketspublic—
Your open orders + trade history + balancesuser_dataprivatetoken (→ user) + market

Two surprises we fixed:

  1. The backend already pushed orderbook and markets (the WSDataFeederJob handled those cases and reused the cached order-book snapshot) — but the front-end's handleMessage only dispatched ohlcv, so those messages were silently dropped. A three-line generic dispatcher fixed the public side, front-end only.
  2. There was no private channel at all. We added one, minimally, by extending the same 1-second feeder rather than inventing new machinery:
// scheduler/ws_data_feeder.go — the new private case (server pushes, client never polls)
case ws.USER_DATA:
    p := key.Params.(ws.PrivateReqParams)
    user, err := W.credentialCache.Get(p.Token)     // token -> user (same cache the REST auth uses)
    if err != nil || user == nil { return }
    openResp,  _ := W.orderService.PaginationQuery(ctx, &dto.GetOrdersQueryReq{UserID: user.ID, Market: p.Market, Type: dto.OPENING_ORDER, ...})
    closedResp,_ := W.orderService.PaginationQuery(ctx, &dto.GetOrdersQueryReq{UserID: user.ID, Market: p.Market, Type: dto.CLOSED_ORDER,  ...})
    balances,  _ := W.balanceService.GetBalances(ctx, user.ID)
    W.wsHub.BroadcastToSubscribers(key, map[string]interface{}{
        "open_orders": openResp.Result, "open_total": openResp.Total,
        "closed_orders": closedResp.Result, "closed_total": closedResp.Total,
        "balances": balances})

Total change for the private feed: 3 constants/structs in ws/model.go, one case in the existing feeder, and one line in the container to inject the extra services. On the front-end, one generic dispatcher plus a subscribeUserData(token, market), and the 1-second polling loop is deleted. Why the token as the key? The client authenticates once at login and holds a token; passing it as the subscription param lets the server resolve the user with the same in-memory credential cache the REST auth uses — no new auth path. Because the push is server-scheduled (once/second, debounced by design), a browser tab makes zero per-second REST calls now; the order book and the user's own orders/trades/balances all arrive over one socket.

What still can't be WebSocket, and why. The push is a fixed 1 Hz snapshot, not a per-event stream. Truly event-driven per-user pushes (fire exactly when your order fills) would mean emitting an event from the order hot path and routing it to your socket — more moving parts, and at hundreds of orders/sec it would hammer the single SQLite writer. The 1 Hz snapshot is the minimal clean design: live enough to watch, cheap enough not to fight the write path.

D.7 — Hosting the whole stack on one VM

For the live demo both halves run on a single machine, reachable from any browser:

  • Backend — the cross-compiled static binary under systemd (:8080 REST, :8081 WS).
  • Frontend — npm run build locally, then the dist/ served by a ~20-line Go static file server (also under systemd, on :80) with an SPA fallback: any unknown path returns index.html so deep links like /markets/BTC-USDT work on refresh. BASE_URL and the WS endpoint in the front-end are set to the VM's IP before building.

Open http://<VM_IP>/markets/BTC-USDT and the page talks to the same box for REST, WS, and static assets. The static server is external tooling (like the load generator) — not part of the application module, so the repo stays clean.
